# go_diary 项目设计文档 (V0.1)

## 项目概述

一个基于 Golang 的微服务架构日记后端项目，用于学习 CI/CD 相关内容。

- **项目名称**: go_diary
- **版本**: V0.1
- **阶段**: Demo
- **目标**: 实现日记的增删改查基础功能，配合 GitHub Actions 学习 CI/CD

## 服务架构

### 核心服务 (go-diary-core)
- 提供日记 CRUD API
- 用户认证（简化版）
- 数据持久化

### AI 总结服务 (go-diary-ai-summary)
- 对日记内容进行 AI 总结（暂不实现）

### 数据库服务 (go-diary-sqldata)
- 通过 Docker Compose 启动 PostgreSQL
- 存储日记数据

## 技术栈

| 组件 | 选型 |
|------|------|
| 语言 | Go 1.21+ |
| Web 框架 | Gin |
| ORM | GORM |
| 数据库 | PostgreSQL |

## 核心服务模块职责

### Handlers 层
- 处理 HTTP 请求
- 参数绑定与校验
- 调用 Service 层
- 返回统一响应格式

### Input Models 层
- 定义请求参数结构
- 包含参数校验标签

### View Models 层
- 定义响应数据结构
- 按需组装返回数据

### Models 层
- 数据模型定义
- 数据库表映射

### Services 层
- 业务逻辑处理
- 定义接口（i_*.go）便于测试和替换
- 提供接口实现
- 依赖注入（通过构造函数）
- 调用数据访问

## 接口设计

### Service 接口定义

```go
// DiaryService 日记服务接口
type DiaryService interface {
    CreateDiary(ctx context.Context, input CreateDiaryInput) (*DiaryViewModel, error)
    GetDiary(ctx context.Context, id int64) (*DiaryViewModel, error)
    ListDiaries(ctx context.Context) ([]*DiaryViewModel, error)
    UpdateDiary(ctx context.Context, id int64, input UpdateDiaryInput) (*DiaryViewModel, error)
    DeleteDiary(ctx context.Context, id int64) error
}

// DiaryRepository 日记仓库接口
type DiaryRepository interface {
    Create(ctx context.Context, diary *Diary) error
    GetByID(ctx context.Context, id int64) (*Diary, error)
    List(ctx context.Context) ([]*Diary, error)
    Update(ctx context.Context, diary *Diary) error
    Delete(ctx context.Context, id int64) error
}

// DatabaseConnectionProvider 数据库连接提供者
type DatabaseConnectionProvider interface {
    GetDB() *gorm.DB
}
```

## 模型设计

### Input Models

```go

    Title   string `json:"title" binding:"required"`

}
```

### Models

```go
// Diary 日记数据模型
type Diary struct {
    ID        int64     `gorm:"primaryKey;autoIncrement"`
    Title     string    `gorm:"size:255;not null"`
    Content   string    `gorm:"type:text;not null"`
    CreatedAt time.Time `gorm:"autoCreateTime"`
    UpdatedAt time.Time `gorm:"autoUpdateTime"`
}
```

### View Models

```go

    Content   string    `json:"content"`

    UpdatedAt time.Time `json:"updated_at"`
}
```

## 依赖注入设计

### 当前实现

当前项目采用**手动依赖注入**的方式，通过构造函数传递依赖：

```go
// 手动创建依赖链
cfg := config.Load()
dbProvider := services.NewDatabaseConnectionProvider(cfg)
diaryRepo := services.NewDiaryRepository(dbProvider)
diaryService := services.NewDiaryService(diaryRepo)
diaryHandler := handlers.NewDiaryHandler(diaryService)
```

### 依赖注入方式说明

1. **构造函数注入**：
   - 每个组件通过构造函数接收其依赖
   - 依赖通过接口定义，便于测试和替换
   - 符合 SOLID 原则中的依赖倒置原则

2. **无 IoC 容器**：
   - 当前实现未使用专门的 IoC 容器
   - 采用手动创建和传递依赖的方式
   - 适合小型项目，结构清晰易懂

### 未来改进建议

对于更大规模的项目，可以考虑引入专门的 IoC 容器：

1. **推荐的 IoC 容器**：
   - **wire** - Google 官方推荐的依赖注入工具
   - **dig** - Uber 开源的依赖注入容器
   - **fx** - Uber 开源的应用框架，包含依赖注入

2. **使用 IoC 容器的优势**：
   - 自动解析依赖关系
   - 减少手动依赖管理的代码
   - 支持生命周期管理（单例、作用域等）
   - 便于测试和模拟依赖

## 依赖注入与测试

- Service 层定义接口，便于 Mock 测试
- 通过构造函数注入依赖（Repository、DatabaseProvider 等）
- 单元测试可使用 Mock 实现替代真实依赖

# go_diary 项目设计文档 (V0.2)

## 项目概述

一个基于 Golang 的微服务架构日记后端项目，用于学习 CI/CD 相关内容。

- **项目名称**: go_diary
- **版本**: V0.2
- **阶段**: Demo
- **目标**: 实现日记的增删改查基础功能，添加用户区分和 UUID 主键，配合 GitHub Actions 学习 CI/CD

## 服务架构

### 核心服务 (go-diary-core)
- 提供日记 CRUD API
- 用户区分（基于 URL 路径）
- 数据持久化

### AI 总结服务 (go-diary-ai-summary)
- 对日记内容进行 AI 总结（暂不实现）

### 数据库服务 (go-diary-sqldata)
- 通过 Docker Compose 启动 PostgreSQL
- 存储日记数据

## 技术栈

| 组件 | 选型 |

| 语言 | Go 1.23+ |
| Web 框架 | Gin |
| ORM | GORM |

## 接口设计

### Service 接口定义

```go
// DiaryService 日记服务接口
type DiaryService interface {
    CreateDiary(ctx context.Context, userIdentity string, input CreateDiaryInput) (*DiaryViewModel, error)
    GetDiary(ctx context.Context, userIdentity string, id string) (*DiaryViewModel, error)
    ListDiaries(ctx context.Context, userIdentity string) ([]*DiaryViewModel, error)
    UpdateDiary(ctx context.Context, userIdentity string, id string, input UpdateDiaryInput) (*DiaryViewModel, error)
    DeleteDiary(ctx context.Context, userIdentity string, id string) error
}

// DiaryRepository 日记仓库接口
type DiaryRepository interface {
    Create(ctx context.Context, diary *Diary) error
    GetByID(ctx context.Context, id string, userIdentity string) (*Diary, error)
    List(ctx context.Context, userIdentity string) ([]*Diary, error)
    Update(ctx context.Context, diary *Diary) error
    Delete(ctx context.Context, id string, userIdentity string) error
}

// DatabaseConnectionProvider 数据库连接提供者
type DatabaseConnectionProvider interface {
    GetDB() *gorm.DB
}

// IdentityService 身份服务接口
type IdentityService interface {
    GetCurrentUserIdentity() string
}
```

## API 接口设计

### 日记接口

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | /:userIdentity/api/v1/diaries | 获取指定用户的日记列表 |
| GET | /:userIdentity/api/v1/diaries/:id | 获取指定用户的单条日记 |
| POST | /:userIdentity/api/v1/diaries | 为指定用户创建日记 |
| PUT | /:userIdentity/api/v1/diaries/:id | 更新指定用户的日记 |
| DELETE | /:userIdentity/api/v1/diaries/:id | 删除指定用户的日记 |

**注意**：`:userIdentity` 只能是 `admin1` 或 `admin2`（模拟用户）。

### 统一响应格式

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### Models

```go
// Diary 日记数据模型
type Diary struct {
    ID               string    `gorm:"primaryKey;type:varchar(36)"`
    UserIdentityGuid string    `gorm:"type:varchar(50);not null;index"`
    Title            string    `gorm:"size:255;not null"`
    Content          string    `gorm:"type:text;not null"`
    CreatedAt        time.Time `gorm:"autoCreateTime"`
    UpdatedAt        time.Time `gorm:"autoUpdateTime"`
}

// BeforeCreate GORM 钩子，创建前生成 UUID
func (d *Diary) BeforeCreate(tx *gorm.DB) error {
    if d.ID == "" {
        d.ID = uuid.New().String()
    }
    return nil
}
```

## 中间件设计

### 用户身份提取中间件

```go
// UserIdentityMiddleware 从 URL 路径提取用户标识
func UserIdentityMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        // 从 URL 路径提取用户标识
        userIdentity := c.Param("userIdentity")
        
        // 验证用户标识是否在白名单中
        if userIdentity != "admin1" && userIdentity != "admin2" {
            response.Error(c, http.StatusForbidden, "Invalid user identity")
            c.Abort()
            return
        }
        
        // 将用户标识存储到上下文
        c.Set("userIdentity", userIdentity)
        c.Next()
    }
}
```

## 路由设计

```go
// 路由分组
userGroup := r.Group("/:userIdentity")

        }
    }

```

## 依赖注入设计

### 当前实现

当前项目采用**手动依赖注入**的方式，通过构造函数传递依赖：

```go
// 手动创建依赖链
cfg := config.Load()
dbProvider := services.NewDatabaseConnectionProvider(cfg)
diaryRepo := services.NewDiaryRepository(dbProvider)
diaryService := services.NewDiaryService(diaryRepo)
diaryHandler := handlers.NewDiaryHandler(diaryService)
```

### 依赖注入方式说明

1. **构造函数注入**：
   - 每个组件通过构造函数接收其依赖
   - 依赖通过接口定义，便于测试和替换
   - 符合 SOLID 原则中的依赖倒置原则

2. **无 IoC 容器**：
   - 当前实现未使用专门的 IoC 容器
   - 采用手动创建和传递依赖的方式
   - 适合小型项目，结构清晰易懂

## 数据库处理

### 实施步骤

1. 停止当前服务：`docker-compose down`
2. 删除数据卷：`docker volume rm go-diary_postgres_data`
3. 重新启动：`docker-compose up -d`
4. 启动应用：`go run src/main.go`

### 优势

- 简单直接，适合学习项目
- 无需编写复杂的迁移脚本
- GORM 会自动根据新模型创建表结构

## 安全性考虑

1. **白名单验证**：验证 URL 中的用户标识是否在允许的列表中（admin1、admin2）
2. **参数验证**：确保用户只能访问自己的日记数据
3. **日志记录**：记录关键操作的用户标识
4. **UUID 主键**：避免递增 ID 的安全问题（可预测性）

## CI/CD 流程

通过 GitHub Actions 实现：
1. 代码提交触发单元测试
2. 构建 Docker 镜像
3. 运行 API 测试
4. 推送镜像（可选）

## V0.1 到 V0.2 的主要变更

1. **用户区分**：添加基于 URL 路径的用户标识
2. **UUID 主键**：将递增 ID 改为 UUID
3. **数据隔离**：用户只能访问自己的日记
4. **中间件**：添加用户身份提取和验证
5. **API 路由**：调整为 `/:userIdentity/api/v1/diaries` 格式
6. **数据模型**：添加 user_identity_guid 字段
7. **服务接口**：所有方法添加用户标识参数
8. **身份服务**：添加 IdentityService 接口和 MockIdentityService 实现

这些变更使项目更加符合实际应用场景，同时保持学习项目的简洁性。
